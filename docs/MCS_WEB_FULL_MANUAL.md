## 1. Overview

MCS Web is the web-based monitoring and control interface for the **Modbus Consolidator System (MCS)**. It is a Node.js/Express application that provides a browser UI and REST API for:

- Defining and managing the device topology (Modbus sources and their replication targets).
- Constructing and compiling the YAML configuration files consumed by MMA2 and the Replicator.
- Controlling MMA and Replicator Docker containers (start / stop / restart).
- Monitoring device operational status via live Modbus reads.
- Viewing and configuring the MMA2 Access Events stream.

### What MCS Web does

- Accepts user input via browser forms.
- Persists the internal model to `model.json`.
- Compiles `model.json` → `replicator/config.yaml` + `mma/config.yaml`.
- Validates generated YAML against MMA2 rules before writing.
- Streams Docker container logs to the browser.
- Issues Modbus FC03 reads to MMA's status registers to report device health.
- Proxies the MMA2 NDJSON Access Events stream to the browser.

### What MCS Web does NOT do

- It does not process Modbus frames at runtime.
- It does not store or replay event history — there is no persistent event log.
- It does not contain Modbus device logic; that belongs to the Replicator.
- It does not modify configuration while services are running — changes take effect only after APPLY.
- It does not validate source-device reachability during configuration (TCP reachability is tested on demand via Test Connection only).

### Observer-only components

The **Access Events** tab displays live NDJSON events proxied from MMA2's HTTP stream endpoint. MCS Web does not alter, buffer, or store these events. If no consumer is connected, MMA2 drops events silently. MCS Web participates only as a passive HTTP client of that stream.

## 2. Architecture

### System topology

```
Browser
  │
  ▼
MCS Web (port 8081 → container port 8080)
  │  - REST API + static HTML
  │  - model.json (internal state)
  │  - compiles → config YAMLs
  │  - Docker socket: starts / stops services
  │  - Modbus TCP: reads status registers
  │  - HTTP: proxies Access Events stream
  ▼
MMA2 (Modbus Memory Appliance)
  │  - Listens on configured TCP ports
  │  - Serves Modbus memory
  │  - Emits Access Events NDJSON stream
  ▼
Physical / virtual Modbus devices
  (read by Replicator, written into MMA2)
```

### Components

| Component | Role |
|-----------|------|
| **Browser (UI)** | Renders tabs; issues API calls to MCS Web backend. |
| **MCS Web backend** (`web/src/index.js`) | Express server. Owns model.json, runs YAML compilation, controls Docker, proxies events. |
| **model.json** | Single source of truth for device topology and memory layout inside MCS Web. |
| `data/mma/config.yaml` | MMA2 configuration — generated by MCS Web. |
| `data/replicator/config.yaml` | Replicator configuration — generated by MCS Web. |
| **MMA2** | External Go service. Serves Modbus TCP memory; runs Access Events. Never modified by MCS Web at runtime. |
| **Replicator** | External service. Polls source devices and writes data into MMA2. |
| **Docker socket** (`/var/run/docker.sock`) | Allows MCS Web to query and control `mma` and `replicator` containers. |

### Read path vs write path

- **Read (status)**: MCS Web issues Modbus FC03 to MMA2's status memory unit every 5 seconds and renders the results in the Status tab.
- **Write (config)**: User edits model → MCS Web compiles YAMLs → user triggers APPLY → configs are written to disk → services restart.

## 3. Version Requirements

| Dependency | Minimum version | Notes |
|------------|----------------|-------|
| MMA2 | `2.3.4` (Docker tag) | Currently deployed as `rodtamin/modbus-memory-appliance:2.3.4`. |
| Docker Engine | Any version with Docker socket API | Required for runtime control. |
| Docker Compose | Plugin form (`docker compose`) | Used for stack management. |
| Node.js | Baked into the `mcs-web` image | Not installed on the host. |

### Version mismatch risks

- MCS Web generates YAML that conforms to the MMA2 configuration contract documented in `docs/mma_manual.md`. Using an older or newer MMA2 image may cause startup failures if the YAML format changes.
- The `mma` image tag in `docker-compose.yaml` is `rodtamin/modbus-memory-appliance:2.3.4`. Pinning to this exact tag prevents accidental format breakage on `docker compose pull`.

## 4. Installation & Deployment

### Prerequisites

- Docker Engine installed and running.
- Docker Compose plugin installed (`docker compose` command available).
- The Docker socket (`/var/run/docker.sock`) is accessible to the `mcs-web` container.

### Directory structure

```
replicator-stack/
├── docker-compose.yaml
├── data/
│   ├── auth.json          # auto-created on first run
│   ├── model.json         # auto-created on first model save
│   ├── mma/
│   │   └── config.yaml    # written by MCS Web on APPLY
│   └── replicator/
│       └── config.yaml    # written by MCS Web on APPLY
```

### docker-compose.yaml (complete example)

```yaml
version: "3.9"

services:

  mma:
    image: rodtamin/modbus-memory-appliance:2.3.4
    container_name: mma
    restart: unless-stopped
    network_mode: "host"
    volumes:
      - ./data/mma:/config
    command: ["/config/config.yaml"]
    environment:
      - APP_VERSION=${APP_VERSION:-dev}
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  replicator:
    image: rodtamin/modbus-replicator:latest
    container_name: replicator
    restart: unless-stopped
    depends_on:
      - mma
    volumes:
      - ./data/replicator:/config
    command: ["/config/config.yaml"]
    environment:
      - APP_VERSION=${APP_VERSION:-dev}
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    networks:
      - replicator-net

  web:
    build:
      context: ./web
      args:
        APP_VERSION: ${APP_VERSION:-dev}
        GIT_SHA: ${GIT_SHA:-}
    image: rodtamin/mcs-web:${APP_VERSION:-dev}
    container_name: mcs-web
    restart: unless-stopped
    ports:
      - "8081:8080"
    volumes:
      - ./data:/app/data
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - DOCKER_DIGEST=${DOCKER_DIGEST:-}
      - APP_VERSION=${APP_VERSION:-dev}
      - GIT_SHA=${GIT_SHA:-}
    depends_on:
      - mma
      - replicator
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    networks:
      - replicator-net

networks:
  replicator-net:
    driver: bridge
```

### Networking

- `mma` runs with `network_mode: "host"` — it binds Modbus TCP ports directly on the host.
- `replicator` and `mcs-web` share `replicator-net` (a bridge network). The Replicator reaches MMA2 via `mma:502` (container DNS on the host network overlay).
- Port `8081` on the host maps to `8080` inside the `mcs-web` container. All browser access is through `http://<host>:8081`.
- The MMA2 Access Events HTTP server (e.g., `:9090`) must be reachable from inside the `mcs-web` container. Because MMA2 uses `network_mode: "host"`, MCS Web accesses it via `TARGET_HOST` (default: `mma`), which resolves to the host's loopback or bridge address depending on Docker networking.

### Environment variables

| Variable | Default | Description |
|----------|---------|-------------|
| `APP_VERSION` | `dev` | Version label displayed in the UI and included in compiled YAMLs. |
| `GIT_SHA` | _(empty)_ | Git commit SHA displayed in the UI build-identity chip. |
| `DOCKER_DIGEST` | _(empty)_ | Docker image digest displayed in the UI. Should be set by the deployment pipeline after image push. |
| `DATA_DIR` | `/app/data` | Mount point for persistent data (model, configs, auth). |
| `TARGET_HOST` | `mma` | Hostname used by MCS Web to reach MMA2 (for status reads and Access Events proxy). |
| `DOCKER_SOCKET` | `/var/run/docker.sock` | Path to the Docker daemon socket. |

### First run

1. `docker compose up -d --build`
2. Open `http://localhost:8081` in a browser.
3. Log in with username `admin`, password `admin`.
4. The UI prompts for a password change on first login (`mustChangePassword: true`).

## 5. Configuration

MCS Web stores configuration in `data/model.json`. This file is never edited manually — it is managed exclusively through the MCS Web UI and REST API. On APPLY, MCS Web compiles `model.json` into two YAML files:

- `data/mma/config.yaml` — consumed by MMA2.
- `data/replicator/config.yaml` — consumed by the Replicator.

### Full YAML example (MMA2 config, as generated by MCS Web)

```yaml
debug: false

listeners:
  - id: "mma_501"
    listen: "0.0.0.0:501"
    memory:
      - unit_id: 1
        holding_registers:
          start: 0
          count: 100
        input_registers:
          start: 0
          count: 50
        coils:
          start: 0
          count: 64
        policy:
          rules:
            - id: "allow-all"
              source_ip:
                - "0.0.0.0/0"
                - "::/0"
              allow_fc: [1, 2, 3, 4, 5, 6, 15, 16]
      - unit_id: 246
        holding_registers:
          start: 0
          count: 60

access_events:
  enabled: true
  mode: rate
  window: 5
  key_fields:
    - src_ip
    - function_code
    - action
    - status
    - port
    - unit
  include_counter: true
  limits:
    max_keys: 1000
    ttl: 30
  output:
    type: http_stream
    path: /events
    listen: ":9090"
```

### MMA configuration fields

MCS Web writes the following MMA2 fields. Field definitions are governed by the MMA2 manual (`docs/mma_manual.md`).

#### Top-level

| Field | Type | Required | Default | Behavior |
|-------|------|----------|---------|----------|
| `debug` | boolean | No | `false` | When `true`, MMA2 logs protocol-level read errors. When `false`, these are suppressed. |
| `listeners` | array | Yes | — | One entry per MMA listener port. |
| `notify` | object | No | _(absent)_ | Global InfluxDB notification adapter. Only included if configured. |
| `access_events` | object | No | _(absent)_ | Access Events configuration. Only included when `enabled: true`. |

#### `listeners[]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Unique identifier for this listener (auto-generated from port number). |
| `listen` | string | Yes | TCP bind address, e.g. `"0.0.0.0:501"`. |
| `memory` | array | Yes | Memory unit definitions for this listener. |

#### `memory[]` (per listener)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `unit_id` | integer | Yes | Modbus Unit ID (0–255). Uniquely identifies a memory instance within a listener. |
| `coils` | object | No | `{ start, count }` — coil area (FC 1, 5, 15). |
| `discrete_inputs` | object | No | `{ start, count }` — discrete input area (FC 2). |
| `holding_registers` | object | No | `{ start, count }` — holding register area (FC 3, 6, 16). |
| `input_registers` | object | No | `{ start, count }` — input register area (FC 4). |
| `state_sealing` | object | No | `{ area: "coil", address: <int> }` — seals memory until coil at `address` is set to 1. Only `"coil"` is valid for `area`. |
| `policy` | object | No | Access control rule list. See Access Control below. |

Memory area fields `start` and `count` are zero-based. `start` is the first address; valid addresses are `[start, start + count)`.

#### Policy rules (per memory unit)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `policy.rules[].id` | string | Yes | Unique rule identifier (non-empty string). |
| `policy.rules[].source_ip` | string[] | Yes | List of IPs or CIDRs. IPv4 and IPv6 supported. |
| `policy.rules[].allow_fc` | integer[] | Yes | List of Modbus function codes this rule permits. Valid codes: 1, 2, 3, 4, 5, 6, 15, 16. |

Policy evaluation: top-down, first-match-wins, default-deny. No policy section → all access allowed.

#### Default policy (MCS Web-generated)

When MCS Web creates a memory unit, it assigns this default policy:

```yaml
policy:
  rules:
    - id: "allow-all"
      source_ip:
        - "0.0.0.0/0"
        - "::/0"
      allow_fc: [1, 2, 3, 4, 5, 6, 15, 16]
```

This allows all clients full access. Modify it in the Memory tab → unit editor to restrict access.

#### Status memory unit

MCS Web auto-generates a reserved memory unit per MMA listener port for device status reporting. This unit:

- Has unit_id = `246` (default; `DEFAULT_STATUS_UNIT_ID` constant in `modelStore.js`).
- Contains only `holding_registers`, starting at 0, sized as `deviceCount × 30` registers.
- Is written and maintained by the Replicator.
- Is read by MCS Web's Status tab polling (Modbus FC03).

This unit is system-managed. Do not define custom devices at unit_id 246 unless you have manually changed the canonical status unit ID.

## 6. Access Events

### Overview

The Access Events system is an observer-only, non-blocking, best-effort mechanism built into MMA2. MCS Web configures it, proxies its stream, and displays events in the Access Events tab. MCS Web has no influence on how MMA2 handles events — it is a consumer only.

Key properties (from MMA2 manual):

- **Observer-only**: observes access control decisions without influencing them.
- **Non-blocking**: event emission never delays Modbus response paths.
- **No persistence**: events are not stored; no replay or history.
- **Best-effort**: drops are expected under load. Slow consumers, full channel buffers, and key map overflow all result in silent drops.

### Full YAML example (access_events section)

```yaml
access_events:
  enabled: true
  mode: rate
  window: 5
  key_fields:
    - src_ip
    - function_code
    - action
    - status
    - port
    - unit
  include_counter: true
  limits:
    max_keys: 1000
    ttl: 30
  output:
    type: http_stream
    path: /events
    listen: ":9090"
```

### Field reference

| Field | Type | Required when enabled | Description |
|-------|------|----------------------|-------------|
| `enabled` | boolean | — | Enables/disables the system. When `false`, no listener starts and all other fields are ignored. |
| `mode` | string | Yes | Only `"rate"` is supported. Any other value causes MMA2 startup failure. |
| `window` | integer | Yes | Aggregation window in seconds. Must be `> 0`. |
| `key_fields` | list | Yes | Must contain exactly these six fields: `src_ip`, `function_code`, `action`, `status`, `port`, `unit`. No substitutions, no omissions. |
| `include_counter` | boolean | Yes | When `true`, summary events include `count` and `window_sec`. |
| `limits.max_keys` | integer | Yes | Maximum concurrent aggregation keys. Must be `> 0`. New keys are dropped when this limit is reached. |
| `limits.ttl` | integer | Yes | Key TTL in seconds. Must be `>= 2 * window`. |
| `output.type` | string | Yes | Only `"http_stream"` is supported. |
| `output.path` | string | Yes | HTTP path for the streaming endpoint. Must start with `"/"`. |
| `output.listen` | string | Yes | TCP bind address for the HTTP server (e.g. `":9090"`). Must not be empty. |

`mode`, `key_fields`, and `output.type` are **read-only** in the MCS Web UI — they are always set to their required values and cannot be changed.

### Behavior scenarios

**Successful observation**

1. MCS Web connects to `http://<TARGET_HOST>:<port><path>` (proxy endpoint).
2. MMA2 streams NDJSON events. The first occurrence of a unique `(src_ip, function_code, action, status, port, unit)` combination in a window emits immediately. Subsequent duplicates within the window are suppressed.
3. When the window expires and there were suppressed events, MMA2 emits a summary event with `count` and `window_sec` (when `include_counter: true`).

**Timeout / MMA2 unreachable**

MCS Web issues an HTTP request to MMA2. If the connection is refused or the upstream returns a non-200 status, MCS Web returns HTTP 502 to the browser. The UI displays "Not connected" in the Access Events Monitor.

**MMA2 busy (exception 0x06)**

Access events operate independently of MMA2's state-sealing mechanism. Modbus exception 0x06 (Device Busy — returned when memory is sealed) is still logged as an access control decision (`status: "denied"`) if it passes through the access control layer. However, MMA2 applies state sealing before the policy layer — sealed memory returns 0x06 before any policy check, so events for sealed-memory requests may not appear.

**At capacity (`max_keys` reached)**

MMA2 drops new keys silently. Existing keys continue to aggregate. MCS Web receives fewer events but continues streaming. No error is signaled.

### Connection flow in MCS Web

1. User opens the Access Events tab → Monitor sub-tab.
2. MCS Web browser makes a `GET /access-events/stream` call.
3. MCS Web backend reads `model.access_events` to find `output.listen` and `output.path`.
4. Backend opens an HTTP connection to `http://<TARGET_HOST>:<port><path>` and pipes the response to the browser.
5. If `access_events.enabled` is `false`, the backend returns 503 immediately.

## 7. UI Features

MCS Web is a single-page application (SPA). All content is in `web/public/index.html`. The application requires login before any tab is rendered.

### 7.0 Authentication

**Login page** (`/login.html`):

- Username and password fields.
- Default credentials: `admin` / `admin`.
- Default credentials force a password change on first login (`mustChangePassword` flag).
- Failed logins return HTTP 401. Rate limit: 20 attempts per 15-minute window per IP.

**Session management**:

- Cookie: `mcs_session` (HttpOnly, SameSite=Strict).
- Session TTL: 24 hours.
- Session is validated server-side on every request.

**Password change** (burger menu → 🔑 Change Password):

- Requires current password.
- New password minimum: 8 characters.
- Hashing: scrypt (N=16384, r=8, p=1).

**Logout** (burger menu → 🚪 Logout):

- Deletes the session server-side and clears the cookie.

### 7.1 Header and global controls

The header is always visible:

- **Build identity chip**: shows `APP_VERSION`, optional `GIT_SHA`, and optional `DOCKER_DIGEST`. Clicking the digest chip copies the full digest to the clipboard.
- **Save button**: appears as "Save" (disabled, gray) when the model is clean. Turns blue with a glow when there are unsaved changes. Clicking it persists the current form state to the backend.
- **Compile status badge**: shows IDLE / COMPILING / OK / ERROR. Updated after every compile triggered by a model mutation (50ms debounce).
- **↻ Refresh button** (in tab bar): manually reloads the model from the backend.
- **Burger menu** (≡): contains Report, Export Config, Import Config, Change Password, Logout.

### 7.2 Model Tab (default tab)

The Model tab is the main workspace for device configuration. It is split into two panels:

#### Left panel: Device tree

- Lists all groups and ungrouped devices.
- Devices are shown with a status dot:
  - **Gray** (`ds-unknown`): no status data yet (before first poll or if poll returned null).
  - **Green** (`ds-ok`): `health_code = 1` (online).
  - **Red** (`ds-error`): `health_code ≠ 1` (offline or in error).
  - **Amber** (`ds-stale`): stale data (internal to the dot state machine; see polling behavior).
  - **Dark gray** (`ds-disabled`): device has no reads configured (red left border on tree row).
- Right-click a device: context menu with Duplicate, Move to Group, Delete.
- Clicking a device: opens the device workspace in the right panel.
- Group header actions: Edit (rename), Delete.
- "＋ Group" button: opens the Add Group dialog.
- "＋ Device" button (in tree header): opens the Add Device dialog.

**Add Device dialog** fields:

| Field | Description |
|-------|-------------|
| Name | Human-readable label. |
| Group | Optional group assignment. |
| Source Endpoint | IP and port of the source Modbus device (e.g. `10.0.0.5:502`). |
| Source Unit ID | Modbus Unit ID on the source device. Non-negative integer. |
| Target Endpoint | MMA2 endpoint where replicated data is stored (e.g. `mma:501`). |
| Target Unit ID | Modbus Unit ID within MMA2 for this device's data. Non-negative integer. |

#### Right panel: Device workspace

Shown when a device is selected. Contains three sections:

**Source section** — editable fields: Name, Group, Source Endpoint, Source Unit ID.

**Reads section** — table of configured Modbus reads for this device.

Each read has:

| Field | Description |
|-------|-------------|
| Name | Human-readable label. |
| Source Area | One of: `holding_registers`, `input_registers`, `coils`, `discrete_inputs`. |
| Source Address | Starting register/coil address (0-based). |
| Count | Number of registers/coils to read. |
| Poll Interval (ms) | Polling interval in milliseconds (minimum 100ms). |

Read actions: Edit, Delete. The "＋ Add Read" form is shown below the table.

**Target section** — editable fields: Target Endpoint, Target Unit ID.

**Test Connection button**: initiates a TCP connection test (`POST /test-connection`). Reports latency in milliseconds on success, or the error reason on failure. Timeout is 3 seconds. Tests only TCP reachability — no Modbus frames are sent.

**Export CSV / Import CSV** (per device):

- Export: downloads the reads as a CSV file (`reads_<deviceId>.csv`).
- Import: uploads a CSV file; rows are validated and inserted. Required columns: `name`, `source_area`, `source_address`, `count`, `poll_interval_ms`.

**Advanced Configuration** (collapsible):

- Override MMA Endpoint: when checked, exposes Host and Port fields that override the global `TARGET_HOST` for this device's MMA connection.

#### CHECK — Integrity Check section

Below the device workspace in the Model tab:

- **🔍 Run Check** button: calls `GET /yaml-integrity` → runs `computeIntegrity(model)`. Reports issues (errors and warnings) by device.
- **🔧 Fix Issues** button: appears when errors exist. Calls `POST /yaml-integrity/fix` to auto-correct `status_slot` and `status_unit_id` inconsistencies.
- APPLY is blocked until the integrity check passes.

### 7.3 Memory Tab

The Memory tab manages the MMA2 memory layout (listeners, units, areas, policy, state sealing).

#### Left panel: Memory port tree

- Lists all configured memory ports.
- "＋ Add Memory Port" form: enter a port number (1–65535), click Add.

#### Right panel: Memory port / unit editor

**Port-level**:

- Port number field (editable).
- Delete port button.
- List of units under the port.
- "＋ Add Unit" form: enter `unit_id` (0–65535), click Add. A default allow-all policy is assigned automatically.
- **Auto-populate** button: calls `POST /memory/port/:portId/units/populate`. Derives units and areas from the current device reads on this port, merging into any existing manual configuration.

**Unit-level** (per unit):

- `unit_id` field (editable).
- State Sealing toggle + address field: when enabled, sets `state_sealing: { area: "coil", address: <int> }`. Only `coil` is supported.
- Access Control (Policy) editor: add/edit/delete policy rules. Each rule has `id`, `source_ip` list, and `allow_fc` list.
- Areas list: shows configured areas (holding_registers, input_registers, coils, discrete_inputs).
- Per area: segments list (start, count, optional name). Add / edit / delete segments. Merge Segments button: collapses all segments into the widest contiguous range.
- Delete unit button.

**Memory Consistency Check** (bottom of Memory tab):

- **🔍 Check Memory Consistency** button: calls `GET /memory/reconcile`. Compares device configurations against the memory registry.
- Reports:
  - **Missing**: devices with no memory allocation.
  - **Area Mismatch**: units missing required area types or address ranges.
  - **Status Memory Missing**: status unit does not exist.
  - **Status Size Mismatch**: status unit has incorrect allocation size.
  - **Orphaned**: units with no corresponding device.
  - **Valid**: fully consistent device/unit pairs.
- Fix actions: individual "Create" or "Fix" buttons per issue category. These call `POST /memory/reconcile/create`, `POST /memory/reconcile/fix-areas`, or `POST /memory/reconcile/fix-status`.

### 7.4 Runtime Tab

The Runtime tab controls Docker containers and exposes live logs.

#### Service status cards

Two cards: **MMA** and **Replicator**.

Each shows:

- Status dot and label: `running`, `stopped`, `not_found`, `unknown` (read from Docker socket via `GET /runtime/status`).
- Buttons: ▶ Start, ■ Stop, ↺ Restart.
- MMA Stop and Restart send a graceful stop timeout of 30 seconds (`?t=30`) to Docker before SIGKILL.

Status is loaded on tab open; manual refresh via **↻ Refresh Status** button.

#### APPLY — Global Actions

Two buttons:

- **⚙ Apply Config**: calls `POST /runtime/apply`. Compiles model → writes YAMLs. Does not restart services.
- **⚙ Apply + Restart All**: calls `POST /runtime/apply-restart`. Compiles model → writes YAMLs → restarts MMA (graceful, 30s timeout) then Replicator.

Both are blocked if the integrity CHECK reports errors.

A progress bar and result message are shown during the operation.

#### Live Logs

- Service selector: MMA or Replicator.
- Log output box: receives streamed Docker container logs via `GET /runtime/logs/:service` (SSE, Docker multiplexed frame format).
- Initial tail: last 100 lines (`DOCKER_LOG_TAIL_LINES=100`).
- **🗑 Clear** button: clears the log box in the browser only; does not affect Docker logs.

### 7.5 Status Tab

Displays live device operational status.

- Opens polling when the tab becomes active.
- Polling interval: **5 seconds** (continuous, started at tab activation, stopped when tab is left).
- `startStatusPolling()` is idempotent — multiple activations do not create multiple timers.
- Data source: `GET /devices/status` → backend issues Modbus FC03 reads to MMA2 status registers.

**Status table** columns:

| Column | Source |
|--------|--------|
| Device Name | From model (or device_name from status registers if set). |
| Health Code | Register offset 0: `1` = online, any other = offline/error. |
| Last Error Code | Register offset 1. |
| Seconds in Error | Register offset 2. |
| Requests Total | Register offsets 20–21 (uint32). |
| Responses Valid | Register offsets 22–23 (uint32). |
| Timeouts Total | Register offsets 24–25 (uint32). |
| Transport Errors | Register offsets 26–27 (uint32). |
| Consecutive Fail Current | Register offset 28. |
| Consecutive Fail Max | Register offset 29. |

**Status dot classes** (also shown in the device tree):

| Class | Color | Condition |
|-------|-------|-----------|
| `ds-unknown` | Gray | No data yet (null register read). |
| `ds-ok` | Green | `health_code = 1`. |
| `ds-error` | Red | `health_code ≠ 1` (any non-1 value, including 0). |

When `GET /devices/status` fails (API error), all devices are marked offline (fail-closed behavior).

Each device's status reads a contiguous block of 30 holding registers from MMA2 starting at `status_slot * 30`.

### 7.6 Config Viewer Tab

Read-only display of the currently compiled YAML configs.

- **Replicator Config** panel: shows `data/replicator/config.yaml`.
- **MMA Config** panel: shows `data/mma/config.yaml`.
- Both panels:
  - **READ-ONLY** badge — no editing.
  - Validation badge: green (valid) or red (errors) after `GET /validate` runs.
  - Search box (Ctrl+F): highlights matching text with match count.
  - **⎘ Copy** button: copies the YAML text to the clipboard.
- **↻ Refresh** button: reloads both configs from disk (`GET /config`).
- On tab open, configs are loaded automatically.

## 8. Data Flow & Timing

### Configuration write path

```
User edits model (UI form)
       │
       ▼
POST /device (or /read, /group, /memory/port, /system, …)
       │  - validates input
       │  - mutates model.json (writeModel)
       │  - schedules compile (50ms debounce)
       ▼
scheduleCompile() fires after 50ms
       │  - runs autoCompile(model)
       │  - validates MMA YAML and Replicator YAML
       │  - writes config files on success
       ▼
APPLY (user-triggered)
       │  POST /runtime/apply  or  POST /runtime/apply-restart
       │  - re-runs compileAndWrite
       │  - restarts services if requested
       ▼
MMA2 + Replicator restart and load new configs
```

### Status polling

- Polling starts when the Status tab is opened.
- `startStatusPolling()` is idempotent.
- Poll interval: 5 seconds.
- Each poll calls `GET /devices/status`.
- The backend batches Modbus FC03 reads by `(target_endpoint, status_unit_id)` — one TCP connection per unique endpoint/unit group.
- Modbus read timeout: 2000ms.
- On poll failure (API error or timeout), all devices in the affected group are marked offline.

### Access Events stream

- MCS Web opens a persistent HTTP connection to MMA2's event HTTP server.
- The connection is established when the Monitor sub-tab is active and `enabled: true`.
- Events are parsed as NDJSON line-by-line and inserted into the event table.
- Maximum table size: 500 rows (oldest rows are discarded).
- The stream connection is closed when the browser tab is closed or the user disconnects.

### Compile debounce

- Every CRUD mutation schedules a compile with a 50ms debounce.
- If multiple mutations arrive within 50ms, only one compile runs.
- The compile status badge reflects the result.
- Background auto-compile writes YAML files to disk but does not restart services.

## 9. Error Handling

### Connection failures (Test Connection)

`POST /test-connection` attempts a TCP connection with a 3-second timeout.

| Error code | Message shown |
|-----------|---------------|
| `ECONNREFUSED` | Connection refused |
| `ENOTFOUND` | DNS resolution failed |
| `ETIMEDOUT` | Connection timed out |
| Timeout (3s) | Connection timed out |

### Modbus read failures (Status tab)

If `modbusReadHoldingRegisters` returns `null` (any TCP error, timeout at 2000ms, or response validation failure), the device's status fields are all set to zero and `health_code = 0` → dot shows `ds-unknown` or `ds-error` depending on prior state.

### Compile errors

When auto-compile or APPLY detects YAML validation errors:

- The compile status badge shows ERROR (red).
- The Config Viewer validation badges show error details.
- APPLY returns HTTP 409 with `errorSummary` and options: `auto_fix`, `ignore`, `cancel`.
  - `auto_fix`: auto-fixes duplicate unit IDs, then re-runs compile.
  - `ignore`: forces compile with `skipValidation=true` (unsafe override — logged as warning).
  - `cancel`: aborts the operation, leaving configs unchanged.

### Integrity check failures

`GET /yaml-integrity` returns issues at two severities:

- **Error**: blocks APPLY. User must run Fix Issues first.
- **Warning**: informational; does not block APPLY.

Issues include: missing `status_slot`, duplicate `status_slot`, inconsistent `status_unit_id` across devices sharing the same target endpoint.

### Access Events stream errors

| Condition | Behavior |
|-----------|----------|
| `access_events.enabled = false` | Backend returns 503; UI shows "Not connected". |
| MMA2 stream port unreachable | Backend returns 502; UI shows connection error. |
| MMA2 returns non-200 | Backend returns 502 with status code and partial body. |
| Client disconnects | Backend destroys the upstream connection silently. |

### Docker API errors

If the Docker socket returns a non-204/304 status for start/stop/restart, the backend returns HTTP 502 with the Docker error message. The Runtime tab displays this error in the result area.

### Rate limiting

- Auth endpoints (`/auth/login`, `/auth/status`, `/auth/change-password`): 20 requests per 15-minute window per IP.
- General API endpoints (`/model/snapshot`, `/config`, `/config/global-export`, `/config/global-import`, `/validate`): 60 requests per minute per IP. Returns HTTP 429 when exceeded.

## 10. Use Cases

### Use Case 1: Basic Monitoring Setup

**Goal**: Configure MCS Web to replicate data from a Modbus device into MMA2 and observe device health.

**Setup**: MMA2 and Replicator running. Source device at `10.0.0.5:502`, unit ID 1, with 100 holding registers.

**Steps**:

1. Open MCS Web at `http://<host>:8081`.
2. Log in.
3. Click **＋ Device** in the device tree.
4. Fill in: Name = `PLC-01`, Source Endpoint = `10.0.0.5:502`, Source Unit ID = `1`, Target Endpoint = `mma:501`, Target Unit ID = `1`.
5. Click Add Device.
6. Select `PLC-01` in the tree. Click **＋ Add Read**.
7. Fill in: Source Area = `holding_registers`, Source Address = `0`, Count = `100`, Poll Interval = `1000`.
8. Click **Run Check**. Verify no errors.
9. Open the Runtime tab. Click **⚙ Apply + Restart All**.
10. Open the Status tab. After ~5 seconds, `PLC-01` should show a green dot if the device is reachable.

**Expected system behavior**:

- MCS Web generates `mma/config.yaml` with a listener on port 501, unit_id 1, holding_registers start=0 count=100.
- MCS Web generates `replicator/config.yaml` with a read route from `10.0.0.5:502` unit 1 to MMA2 unit 1.
- Replicator begins polling every 1 second. Health code 1 → green dot in Status tab.

**Setup**: Unsure if `192.168.1.100:502` is online.

### Use Case 3: Access Event Observation

**Goal**: Monitor which Modbus clients are being allowed or denied by MMA2's access control layer.

**Setup**: MMA2 running with access events enabled in the MCS Web configuration.

**Steps**:

**Expected system behavior**:

- Events appear in the table. High-frequency access from the same client/FC/port/unit is aggregated per window; summary events appear with a Count field.
- If the browser tab is closed, the stream connection is dropped. MMA2 discards subsequent events until a consumer reconnects.

**Goal**: Diagnose why a device shows red in the Status tab.

**Steps**:

## 11. Limitations

### No event persistence

Access events are not stored by MMA2 or by MCS Web. If the browser connection is closed, events emitted during the disconnected period are lost.

### No persistent Modbus memory across restarts

MMA2 memory is volatile. Every restart clears all Modbus registers. The Replicator repopulates memory from source devices after restart. Until the first poll completes, all MMA2 memory is zero.

### MMA2 dependency

MCS Web cannot operate usefully without a running MMA2 instance:

- Status tab shows all devices offline.
- Access Events Monitor cannot connect.
- Config Viewer shows the last compiled YAML (from disk), but it is not confirmed as running.

### Docker socket requirement

Runtime control (start / stop / restart) requires access to the Docker socket. If the socket is unavailable, the Runtime tab's service control buttons will fail with a 500 error. Log streaming will also be unavailable.

### Single user

MCS Web supports a single username/password pair. There is no role-based access control. All authenticated users have full access to all operations.

### No real-time config reload

MMA2 loads configuration once at startup. Config changes have no effect until services are restarted. Calling APPLY without restarting writes files to disk but does not change MMA2's running state.

### Overlap validation is source-area-scoped

Read overlap detection checks for address range collisions within the same source area (e.g., two holding_registers reads on the same device that overlap). It does not check cross-device collisions at the MMA2 level — that is enforced during YAML compilation.

## 12. Best Practices

### Keep MMA2 as the source of truth for Modbus memory

Define memory in MCS Web's Memory tab to match exactly what MMA2 needs. Do not manually edit `data/mma/config.yaml` — MCS Web overwrites it on every APPLY.

### Always run CHECK before APPLY

Use **🔍 Run Check** to verify integrity before clicking APPLY. This prevents applying configurations that will cause MMA2 or Replicator to fail at startup.

### Use Auto-populate for initial memory layout

After adding devices and reads, use the **Auto-populate** button in the Memory tab to derive units and areas from device reads. Then refine with state sealing or custom policies as needed.

### Restart order: MMA first, then Replicator

**Apply + Restart All** enforces the correct order: MMA2 stops and restarts first (allowing open Modbus connections to close gracefully within 30 seconds), then the Replicator restarts. Never restart the Replicator before MMA2 — the Replicator will attempt to write to an unavailable memory target and may log errors until MMA2 is ready.

### Set poll intervals based on source device capability

Do not set poll intervals below what the source device can handle. A device that cannot respond within the poll interval will accumulate timeouts (`timeouts_total`) and eventually mark as offline (`health_code ≠ 1`). Minimum allowed poll interval in MCS Web: 100ms.

### Export config before major changes

Use **⬇ Export Config** (burger menu) before making significant topology changes. The exported `global-config.json` file can be imported to restore the model if needed.

### Use the Memory Consistency Check after adding or removing devices

After any device addition or deletion, run **🔍 Check Memory Consistency** in the Memory tab to detect orphaned or missing memory units, and use the Fix actions to correct them.

## 13. Troubleshooting

### MMA2 not reachable (status tab all gray/red)

**Symptom**: All devices show gray (`ds-unknown`) or red (`ds-error`) in the Status tab immediately after startup.

**Diagnosis**:

1. Open the Runtime tab. Check the MMA status dot.
   - If `not_found`: MMA container does not exist — run `docker compose up -d`.
   - If `stopped`: click **▶ Start** or run `docker compose start mma`.
   - If `running` but status tab still shows offline: MMA2 may not have loaded the config yet (check Live Logs for startup errors).
2. Open Live Logs → MMA. Look for startup errors (invalid YAML, port binding failures).
3. Open Config Viewer → MMA Config. Check the validation badge. If red, fix the reported errors and re-apply.
4. Verify that `data/mma/config.yaml` exists (it is created on first APPLY; if missing, run APPLY).

### No events in Access Events Monitor

**Symptom**: Monitor sub-tab shows "Not connected" or the table remains empty.

**Diagnosis**:

1. Open Access Events → Settings. Confirm `enabled` is on.
2. Confirm settings have been saved and APPLY has been run since the last settings change.
3. Open Live Logs → MMA. Look for a line like `access events: listening on :9090`. If absent, access events did not start (check for bind failures).
4. Verify that the `output.listen` port is reachable from inside the `mcs-web` container. If MMA2 uses `network_mode: "host"`, the port should be accessible via `TARGET_HOST` (default: `mma`).
5. If the port is bound but events still do not appear: confirm that Modbus clients are sending requests to MMA2 — no requests means no events.

### Status not updating

**Symptom**: Status tab shows stale or zero values for an otherwise-running device.

**Diagnosis**:

1. Confirm the Status tab is active (polling only runs while the tab is open).
2. Open Live Logs → Replicator. Check for Modbus write errors — the Replicator writes status registers; if it cannot reach MMA2's status unit, values remain zero.
3. Run **🔍 Check Memory Consistency** in the Memory tab. Check for status memory issues (`status_missing` or `status_size_mismatch`) and fix them.
4. Verify `status_slot` and `status_unit_id` are consistent via **🔍 Run Check** in the Model tab. Run **🔧 Fix Issues** if errors are reported.
5. Re-apply after any memory or integrity fixes.

### Config mismatch / APPLY blocked

**Symptom**: APPLY returns an error or is blocked by integrity issues.

**Diagnosis**:

1. Run **🔍 Run Check**. Review the list of errors by device.
2. Click **🔧 Fix Issues** to auto-correct `status_slot` and `status_unit_id` problems.
3. If errors remain after Fix Issues, review each device's configuration for missing reads, invalid endpoints, or duplicate unit IDs.
4. Open Config Viewer → MMA Config and check the validation badge. If red, the YAML has structural errors (duplicate unit IDs, invalid field values). Use `auto_fix` when offered on the APPLY error dialog.
5. If all else fails and you must force an apply despite warnings: select `ignore` on the APPLY error dialog. This bypasses validation — use only as a last resort and check MMA2 startup logs immediately afterward.

### Login fails / session expired

**Symptom**: Cannot log in, or the UI redirects to the login page.

**Diagnosis**:

1. Confirm username is `admin` (single account; no other usernames).
2. If the default password was changed and is now unknown: access the host running MCS Web, delete `data/auth.json`, and restart the `mcs-web` container. This resets credentials to `admin`/`admin` with `mustChangePassword: true`.
3. Session expiry is 24 hours. Re-login is required after expiry.

### Docker socket errors

**Symptom**: Runtime tab shows 500 errors on start/stop/restart; log streaming fails.

**Diagnosis**:

1. Confirm `/var/run/docker.sock` is mounted in `docker-compose.yaml` under the `web` service volumes.
2. Confirm the user inside the container has permission to access the Docker socket (typically by being in the `docker` group or running as root).
3. Restart the `mcs-web` container: `docker compose restart web`.
